# The Journey — Professional Audit Report

**Date:** 2026-06-19
**Audit type:** read-only; no code or files changed during the audit.
**Scope:** clinical canon (`docs/journey/00-shared-core.md` + 8 stage specs), runtime prompts (`journey-master.md`, `stage-01.md`, `stage-02.md`), runtime code (`lib/journey/**`, `app/api/journey/turn/route.ts`).
**Audit question:** is The Journey runtime faithful to the clinical method, is the library written in a way the AI can follow, and are there silent bugs or gaps between intent and implementation?

---

## 1. Executive summary

1. **The clinical canon is excellent and unusually well-written for an LLM.** Eight stage specs + a Shared Core present a coherent, non-regressive, anchor-led method. Decision rules, prohibitions, captures and gates are concrete. As an LLM-target document set, it is among the strongest reviewed.
2. **The active source of truth is the master runtime prompt, not the canon.** `assembleSystemPrompt` (`lib/journey/prompts/assemble.ts:177-202`) sends the AI only `journey-master.md`; the 9 clinical specs are never injected. Drift between canon and master is therefore drift between *what Julia signed off* and *what runs*. There is real drift today.
3. **The master prompt covers Stages 1, 2, 5 and 8 well; Stages 3, 4, 6, 7 are THIN; Stage 4's MII safeguards are largely MISSING from the prompt.** The non-negotiable clinical safeguards (Adult Self leads, 48-hour settling, Securing the Part ritual, three-layer pacing inside Stage 4) live in the canon but not in the prompt the AI actually receives.
4. **The state-report contract has multiple silent failures.** The AI is told to emit fields (`mii6Check`, `softWhyResponse`, `emotionNamed`, `emotionBodyLocation`, `clinicalRead`, `safetyReorientation`, `selfLoyaltyStatement`, `oneSmallAction`, `partSecuredAt`, `bridgeAchievedAt`) that the schema does not accept and the parser silently drops. Several gates check signals that the prompt never tells the AI to emit.
5. **Two CRITICAL safety enum mismatches.** The keyword scanner and verifier emit `panic_severe` / `dissociation_severe` / `flashback_in_progress`; the legacy stage prompts and `assemble.ts`'s output-format instruction tell the AI to emit `panic` / `dissociation` / `flashback`. The freeze reason and downstream review surfaces lose the type for AI-originated freezes that follow `stage-01.md`. The master prompt itself uses the correct suffixed names; this is a partial fix that remains broken on the fallback path.
6. **Stage gates reference state shape that no code populates.** MII-5 in `checkStage4Gate` checks `currentRestingPlace OR any state report carries an "offering" string` — but for `offering` it tests `adultSelfQualities`, which is a Stage 3 capture, not a Stage 4 offering. MII-6 checks `state.mii.mii6_noDestabilisation.status === 'failed'`, but no code path ever writes that field. MII-7 needs `cohesionAwareness` — the master prompt never asks for it.
7. **The "one master prompt" pivot is defensible — but is currently under-resourced.** As a clinical idea (clinician picks the right move regardless of stage) it is sound. As executed, the master prompt is too compressed on the high-risk stages (3, 4, 5, 6, 7) and is silently fighting the gates the code enforces. The pivot can work, but needs an additional ~2,000 tokens of per-stage guardrails in the same file.

## 3. Source-of-truth analysis

There are three potential sources. The active source is **the master runtime prompt**.

- `assembleSystemPrompt` (`lib/journey/prompts/assemble.ts:183-186`) calls `loadMasterJourneyPrompt()` first; if present (it is), it returns the master, injecting only the runtime state block. The 9 clinical specs (`00-shared-core.md` … `08-stage-embodiment.md`) are **never** loaded into the AI's context at runtime when the master exists.
- The legacy per-stage engineered prompts (`runtime/stage-01.md`, `stage-02.md`) are only used as a fallback if the master is absent. They are currently unreachable in production.
- The clinical specs themselves (`docs/journey/00-shared-core.md` etc.) function as **the reviewable document Julia signs off**, but they are reference docs for humans only. The AI does not see them.

**Where they agree:** the AI identity, the voice rules, the Red Flag protocol (almost verbatim), the practice-generation algorithm and families, the agency framing on foreign material, the "no diagnosing third parties" rule. The Shared Core's spine is preserved.

**Where they disagree (drift):**

1. **Stage 3 reproducibility.** Canon §10 requires Adult Self reached on at least two different days; gate enforces this. Master prompt move #3 does not mention reproducibility, so the AI may report `adultSelfPresent: true` and recommend advance without understanding the day-distinct gate. This produces user-invisible churn — the AI says "advance," the code says "stay," nothing tells the AI why.
2. **MII safeguards.** Canon §10 in Stage 4 names seven criteria explicitly. Master prompt move #4 says "one part at a time. Do not open many parts in one turn. The user can fragment." That's it. The Compassion Bridge as a *quality* (`compassion / curiosity / acceptance / willingness_to_comfort`) — which the gate checks — is not in the master prompt's move-4 description, though the field name appears in `<output_format>`.
3. **48-hour Deep Layer guideline.** Canon Stage 4 §10 (MII-6) and Stage 5 §10 require a soft check-in. The code injects this on the next turn (`delayedCheck/signal.ts`), and the injection asks the AI to capture `mii6Check`. But **`mii6Check` is not in the master prompt's `<output_format>` schema and the parser does not read it.** The injected instruction asks for a field that will be silently dropped.
4. **Stage 7 Safety Reorientation.** Canon §8.3 is mandatory at every Stage 7 session close. Master prompt does not mention it at all.
5. **Stage 8 Discharge Protocol.** Canon §8.3 is the six-step closing. Master prompt does not encode it. Yet the `<output_format>` allows `recommendedAction: "discharge"` and the router will set `dischargedAt` on first occurrence.
6. **Practice family of Block 2's "Soft Why" landing on self-attack.** Canon (`02-stage-pain.md` §8.2 watch-for) instructs witnessing self-criticism with `compassion` family. Master prompt allows this only implicitly. Legacy `stage-02.md` (correctly) makes this explicit.

**Bottom line:** the master prompt is governing. The canon is descriptive of intent. The runtime code enforces gates against an idealised state-report shape the master does not fully ask the AI to emit. This is the central architectural risk.

## 4. Clinical library clarity for AI consumption

The clinical specs were written by a clinician. Read at face value, they are clearer for an LLM than for most clinicians, because every spec follows the same skeleton (Stage Purpose / When Active / Methods / Constraints / Depth Permissions / Forbidden Moves / Client-Type Adaptations / Signature Practices / Watch-For Markers / Completion Criteria / Worked Examples).

**Strengths for LLM consumption:**

- **Decision rules are precise.** §10 Completion Criteria in every stage lists boolean predicates: "anchorText is set", "adultSelfPresent true in at least 4 of last 5", "compassionBridgeQuality is set on at least 2 separate sessions". An LLM can mirror these into state-report emissions if asked to.
- **Each forbidden move is concrete.** "Do not name the part for the user", "Do not enter past scenes in sensory detail", "Do not initiate Deep Layer without explicit stability and Adult Self present." These are not abstract principles.
- **Worked examples in §11 give the LLM behavioural exemplars.** This is by far the most LLM-friendly section style. The Stage 3 Example C ("the soft one with the dog") is worth more than the abstract Stage 3 §4 constraints.
- **Voice references are flagged as references, not scripts.** Excellent.

**Where an LLM is likely to misinterpret or hallucinate:**

1. **Stage transitions framed as "the user does not know they are in Stage X".** True clinically; but the LLM is asked to *behave* differently in different stages. The combination of "the user doesn't know" + "your behaviour changes stage to stage" creates a subtle pressure for the LLM to handle stage logic itself, when in fact the code holds it.
2. **Two-different-days requirements (Stages 3, 6, 7).** An LLM has no built-in notion of "session" vs "turn" vs "calendar day". Without code surfacing day-count signals to the AI, it will infer reproducibility from conversational density. The current state block surfaces `lastActivityAt` only implicitly; no day-count is visible.
3. **MII-6 as "settling signal not lockout".** The spec is careful, but the words *"48-hour rule is real"* and *"non-negotiable"* both appear within Stage 4. The LLM may interpret "the rule is real" as a hard block, contradicting the master prompt's "the rule is the AI's gentleness, not a code-blocked timer." The pivot was correct; the canon now needs one sentence cleanup so the LLM doesn't see two registers.
4. **"Compassion Bridge Quality" as an enum.** An LLM will produce paraphrases — "warmth", "presence", "kindness" — instead of the four exact strings (`compassion / curiosity / acceptance / willingness_to_comfort`). The master prompt does not surface the enum to the AI; the parser silently rejects everything else.
5. **"Adult Self present" is binary in the schema but graded in the canon.** Canon talks about Adult Self being "a little wiser, a little steadier, a little kinder", "faint", "close". Schema asks for a boolean. An LLM will under-report adultSelfPresent because the felt experience is partial; the gate then never closes.
6. **Stage 7 vs Stage 8 boundary.** Both stages say "no major life decisions". The cue separating them is whether the user is "embodying in small moments" vs "mapping". This will be hard for an LLM to distinguish reliably without a clear behavioural marker in the state block.

**Most likely hallucinations:**
- Inventing a part the user did not name (Stage 4).
- Naming the Compassion Bridge quality before the user has shown it (Stage 4).
- Producing a "discharge" recommendedAction prematurely when a user says "I feel like myself" once (Stage 8).
- Emitting `mii6Check: "stable"` to clear the soft check-in, when the field is dropped on parse.

## 5. Methodology encoding correctness

The persistent landscape (`JourneyState` in `lib/journey/state/types.ts`) encodes the canon's §8 well: anchor + identityAnchor (separate, never overwritten), processingChannel, adultSelfQualities, parts (with channel, safeDistance, compassionBridgeQuality, currentRestingPlace), foreignFiles (with originDescription, returnedTo, honouringPhrase, whatStaysAsMine, identifiedAt, releasedAt), signatureImages, MII state, continuityNote. This is a faithful schema-level match to the canon.

The Personal Anchor protocol is encoded correctly: `save.ts:70-73` is `set-once and never overwritten`. The Identity Anchor is added alongside, not replacing (`save.ts:74-77`).

The Red Flag protocol is implemented end-to-end with the canonical verbatim string in EN and RU, locale fallback to EN (`keywords.ts:138-164`), pre-LLM keyword scan + post-LLM verifier net, idempotent freeze (`freeze.ts:29-58`), and frozen-state pass-through that still scans incoming messages so an escalating frozen user produces audit rows (`route.ts:114-132`). This is solid clinical engineering.

The Practice Generation Algorithm: the 5 families and the 7-step order from Shared Core §5.2 are reproduced in both the master prompt's `<practice_generation>` and `stage-01.md`'s. The runtime persists `practiceRun` and writes a child `JourneyPracticeRun` row (`audit/log.ts:36-55`). Correct.

**Encoding errors:**

- **MII-5 (basic reparenting capacity) is wired to the wrong field.** `stage-gates.ts:185-188`:
  ```ts
  const reparenting =
    state.parts.some((p) => p.currentRestingPlace) ||
    turns.some((t) => typeof t.report.adultSelfQualities === 'string');
  ```
  The fallback to `adultSelfQualities` is a Stage 3 capture, not a Stage 4 reparenting offering. This trivially passes for any user who reached Stage 3, regardless of whether the Adult Self has offered anything to a part. The canon specifies an `adultSelfOfferingToPart` field in user's words; the schema has `partSecured.adultSelfOffering`; the gate should test that or `state.parts.some(p => p.currentRestingPlace || p.adultSelfOffering)`.

- **MII-6 (no delayed destabilisation) is dead code.** `stage-gates.ts:195` reads `state.mii.mii6_noDestabilisation.status`, but no code path writes to `state.mii.mii6_noDestabilisation`. The injected settling-signal instruction (`delayedCheck/signal.ts:67`) asks the AI to emit `mii6Check`, but (a) `mii6Check` isn't in the schema, (b) the parser doesn't read it, (c) even if it did, nothing would translate it to `mii.mii6_noDestabilisation.status`. **The 48-hour stability gate cannot fail.** A Stage 4 user could complete MII-6 by default.

- **MII-7 (internal cohesion awareness) is asked from the canon, but the master prompt never prompts for `cohesionAwareness`.** The field appears in `assemble.ts:67` and the parser handles it (`parse.ts:168`). But the master prompt's move-4 description doesn't reference it, and the `<output_format>` lists `cohesionAwareness` only as a stage-marker. The AI is unlikely to emit it.

- **Stage 6 cohesion gate uses readinessTouched tokens not in any prompt's vocabulary.** `stage-gates.ts:240-246` searches for `/feel[_-]?like[_-]?myself|internal[_-]?consensus|cohesion/i` in readinessTouched. The master prompt's listed vocabulary is `"anchor_identified", "body_located", "emotion_named", "orientation_present", "pain_named", "alliance_formed", "observer_seat_touched", "adult_self_present", "foreign_file_identified", "foreign_file_released", "formulation_confirmed"`. None of those match. **Stage 6 cannot advance** without the AI guessing a token not in its vocabulary.

- **Stage 6 "no separated parts" check is shape-mismatched.** `stage-gates.ts:250-252` requires `state.parts.every(p => p.compassionBridgeQuality != null)`. But `compassionBridgeQuality` is set per Stage 4 MII-4. A user who hasn't run a Compassion Bridge for every part will fail this — even legitimately. And a user with zero parts in landscape passes vacuously. This is back-to-front.

- **Stage 8 discharge gate**: `stage-gates.ts:319-322` requires "at least 6 CAL sessions on different real moments" via `calRunOn` + distinctDays. The master prompt asks the AI to set `calRunOn` and `calLayer`, which is good. But the AI is never told that 6 distinct days are required, nor that 3 of them must be Layer ≥ 2. A user can reach Stage 8, run CALs across 3 days, and never advance — and no one will know why.

- **`deepLayerContact` is over-triggered.** `save.ts:44` sets `lastDeepLayerContactAt = now` any time `practiceRun.depth === 'deep'`. The canon reserves "Deep Layer" specifically for Stage 4 re-parenting and Stage 5 symbolic release. A Stage 7 Symbolic Identity Map run with `depth: "deep"` will incorrectly trigger the 48-hour settling cue (which is harmless but odd — the canon says Stage 7 doesn't need settling).

- **`parts.every(...)` in Stage 6 will pass vacuously for users with no parts.** A user who reaches Stage 6 with `state.parts.length === 0` (possible if all parts work happened verbally without being captured) passes both Stage 4 MII-2 (`reasons.push('mii2_no_parts_recognised')` — actually this catches it) and Stage 6's part check. Stage 4 catches the empty array; Stage 6 does not.

## 6. Bugs and gaps

### CRITICAL

1. **redFlagType enum mismatch on the legacy path (`stage-01.md`, `assemble.ts` fallback instruction).** `keywords.ts:17-24` exports `panic_severe | dissociation_severe | flashback_in_progress`. The freeze reason composer (`freeze.ts:65-66`) writes `type:${redFlagType}` into the stored reason. The verifier (`verifier.ts:127-128`) and master prompt (`journey-master.md:162`) emit the *correct* `_severe` / `_in_progress` suffixes. But `stage-01.md:113` and `assemble.ts:43` both list `"panic" | "dissociation" | "flashback"`. The parser (`parse.ts:109`) takes any string. If the AI follows the fallback instructions and emits bare `"panic"`, the freeze reason composer stores `type:panic`, which is meaningless to downstream review. Worse, on the verifier's whitelist (`verifier.ts:164-172`) `"panic"` is not present, so cross-checking will reject it. The master prompt path is fine; the legacy path is broken and silently degrades the freeze reason quality.

2. **MII-6 (the 48-hour stability gate) cannot fail.** Detailed above. The clinical "non-negotiable" gate that protects users from premature Deep Layer cascades **does not exist in runtime.** The next-session check-in instruction asks for `mii6Check`, the parser ignores it, `state.mii.mii6_noDestabilisation` is never written, the gate's only fail condition (`mii6Status === 'failed'`) is unreachable.

3. **Stage 6 advancement gate is unreachable.** The token vocabulary the gate searches for (`feel_like_myself`, `internal_consensus`, `cohesion`) is not in any prompt's readinessTouched vocabulary. A perfect Stage 6 user reading the master prompt would never produce a matching token. The route would log `felt_cohesion_not_reached_on_two_days` forever.

### HIGH

4. **Stage 1 first-message inconsistency.** `stage-01.md` (legacy) instructs: *"Hello. Good to have you here. How are you doing today?"* as the AI's first message. The master prompt (`journey-master.md:449`) says: *"the UI shows a welcome message at the start of every new conversation… You do NOT see the welcome — but you know it was shown. The user has already been greeted. You do not greet them again. You respond directly to what they brought."* These are opposite instructions for the first turn. If the legacy prompt was ever revived as fallback, the user would be double-greeted.

5. **`partSecured` saving partially corrupts the landscape.** `save.ts:153-171`: if the AI emits `partSecured.adultSelfOffering` without `partSecured.restingPlace`, the code stores the offering text *into the part's `currentRestingPlace` column*. This conflates two different fields. Later, `loadJourneyState` will surface the offering as `currentRestingPlace`, and the Adult Self's offering becomes the part's resting-place description.

6. **Schema does not accept fields the engineered prompts tell the AI to emit.** Lists (not exhaustive):
   - Master prompt example 6: `clinicalRead` (used in every example) → not in `StateReport` type, `parse.ts` does not copy it, schema (`schema.ts`) does not declare it. Silently dropped every turn.
   - Legacy `stage-02.md` line 285: `emotionNamed`, `emotionBodyLocation`, `softWhyResponse` → not in `StateReport`, not parsed. Silently dropped.
   - Master prompt §10: `safetyReorientation` (mentioned in canon Stage 7 §8.3) → not in schema, not parsed. Gate cannot verify it ran.
   - Canon Stage 6 §8.3: `selfLoyaltyStatement`, `oneSmallAction` → not in schema, not parsed. Stage 6 gate uses readinessTouched proxy that doesn't fire.
   - Canon Stage 4 §8.2: `bridgeAchievedAt` → not in schema.
   - Canon Stage 4 §8.3: `partSecuredAt`, `partRestingPlace`, `userGrounded` → only `partRestingPlace` is partially supported via `partSecured.restingPlace`.
   - Canon Stage 5 §8: `foreignMaterialIdentified`, `originIdentified`, `burdenReturnedTo`, `somaticRelease`, `bodyConfirmation` → none of these field names are accepted. The mapping is via `foreignFilesTouched[].description`, `foreignFileReleased.{description, returnedTo, honouringPhrase, whatStaysAsMine}`. The AI following the canon will emit the wrong field names.

7. **`emotion_named` not enforceable.** `checkStage2Gate` (`stage-gates.ts:113-118`) accepts `/emotion[_-]?named|emotion[_-]?located|soft[_-]?why/i` from readinessTouched. The master prompt's listed vocabulary includes `"emotion_named"` and `"pain_named"`. So far so good. But for Soft Why, the master prompt vocabulary lists no `soft_why_asked` or `soft_why_answered` tokens — those exist only in `stage-02.md`. A master-prompt AI will never trip the soft-why branch of the regex; the `emotion_named` token alone passes the gate.

8. **`safetyNoneForLast` requires N turns of history to even be possible.** `history.ts:105-109`: `if (tail.length < n) return false`. For Stage 4 with `n = 5`, the gate cannot pass until at least 5 audit turns exist. This is fine for established users but combined with intensity reads above 5 (early Stage 4 users often hit 6+), the gate may take a long time to first pass. Not a bug — but combined with the unreachable MII-6 and 7 fields above, the user is effectively pinned at Stage 4.

9. **Stage 5 `practiceRun.depth === 'deep'` overuse.** Per the canon Stage 5 §10 the symbolic-release gate requires `somaticRelease: true`. The runtime gate (`stage-gates.ts:218-227`) substitutes `state.foreignFiles.some(f => f.releasedAt != null)`. The AI sets `releasedAt` via emitting `foreignFileReleased`, but `somaticRelease` boolean confirmation is not captured. If the AI emits `foreignFileReleased` without an actual somatic release having landed, the gate passes. This trades one criterion for the existence of an emit. Workable but not what canon prescribes.

10. **`finaliseTurn` race: state report saved AFTER router decision.** `route.ts:284-373` — `applyStateReportToProgress` and `runJourneyVerifier` run in parallel via `Promise.all`. Then verifier verdict is computed, then `writeAuditTurn`, then a *fresh reload* of state, then `decideRoute`. The fresh reload after `applyStateReportToProgress` is correct — but `writeAuditTurn` writes the final report which the gates then read via `loadRecentTurns`. The audit row is in by the time the route decides. So gates see the just-completed turn. ✅ This is correct.

### MEDIUM

11. **`urgencyMarkers` discovery path is conversational only.** Stage 7's non-negotiable urgency gate (`stage-gates.ts:289-291`) tests `t.report.urgencyMarkers === 'present'`. The master prompt move-7 description does not tell the AI when to emit `urgencyMarkers: "present"`. The canon Stage 7 §8.3 Watch-for-markers describes urgency markers behaviourally but the master prompt's `<output_format>` lists the field without a behavioural rule. The AI will under-emit. False negatives on urgency = user advances to Stage 8 while still planning to leave their husband on the spot.

12. **`continuityNote` is encrypted as a string but is the AI's running case formulation.** Encrypted at rest, decrypted on each load and injected into the state block. Format is unstructured prose. Two risks: (a) the AI re-writes it each turn ("revise additively"), and slight drift each turn produces inconsistency across sessions. (b) The `<memory>` instruction tells the AI "never wipe history; refine it." Trusting an LLM with that across 30 turns of history compression is real but acceptable. Monitor in production.

13. **Localised crisis numbers.** Canon §7: "Localised crisis numbers are loaded per language at runtime." Runtime: only EN and RU exist (`keywords.ts:138-154`); 6 other locales fall back to EN. Canon promise is unmet for FR/DE/ES/IT/PL/PT.

14. **`adultSelfPresent` is asked as boolean; canon's gate then expects 4-of-5 boolean trues across last 5 turns (`stage-gates.ts:159-160`).** The AI is told this is binary (e.g. `journey-master.md:614`), but canon Stage 3 describes the Adult Self being faint, partial, etc. The binary read will under-estimate presence. Combined with the "4-of-5" requirement, MII-1 will be hard to land for users whose Adult Self is real but unsteady.

15. **No turn-by-turn audit of router decisions.** `applyRouteDecision` mutates `recodeProgress` but writes nothing to the audit log. A stage transition (advance, regress, discharge) is the highest-stakes event in the system — it should produce a dedicated audit row. Today it produces an updated row but no narrative.

### LOW

16. **The `<state-report>` regex partial-tag-prevention buffer is 20 chars.** `route.ts:236`: `STATE_REPORT_OPEN.length` = 14. The buffer holds back `Math.max(0, buffer.length - 14)`. So the last 14 characters are not streamed. Correct, but if the model emits any other text containing the substring `<state-report>` inside a quoted clinical example, it would be truncated. Real risk: low.

17. **Stage gate guards use `last?.report.recommendedAction !== 'advance'` (`stage-gates.ts:53-56`).** If the AI ever emits `recommendedAction: "discharge"` while in Stage < 8, the guard treats it as not-advance, and the user stays. Symmetric for `regress_to_*` (handled correctly via the early-return). This is conservative and safe.

18. **`processingChannel` is overwritten on every turn that reports one.** Save (`save.ts:78`) sets unconditionally. Canon §8: channel "refined over time", which suggests merge logic. Today's behaviour: last-wins. Likely benign.

19. **Cache invalidation.** `load-spec.ts` caches `loadMasterJourneyPrompt` for the process lifetime. A prompt update requires a redeploy. Acceptable.

## 7. The fundamental architectural question — the master-prompt pivot

The plan called for `stage-01.ts` … `stage-08.ts`. The implementation pivoted to one master prompt. Honest read:

**The pivot is sound *as a clinical idea*.** Real clinicians do not run a fixed curriculum. They read what's in the room and reach for the move that serves now. A user who reached Stage 6 may legitimately need anchor work mid-session; a per-stage prompt would have made that awkward. The clinical canon itself acknowledges this in `00-shared-core.md` and across every stage's "regression preserves the landscape" framing.

**The pivot is sound *as engineering*.** Per-stage prompts would have required nine large prompts maintained in dual-source-of-truth with nine clinical specs — 18 documents, all needing to stay aligned. That is a maintenance nightmare. One prompt + one canon is simpler.

**But as currently executed, the pivot under-delivers on Stages 3–8.** The master prompt is ~3,200 tokens. Three of those stages (4, 6, 8) carry the highest clinical risk. Their canonical safeguards (MII, Securing the Part, Discharge Protocol, Safety Reorientation, Internal Consensus Check) are either summarised in a few words or absent. The AI has the *direction* of each move but not the safeguards. Combined with the bugs above — state-report fields the AI emits but the schema silently drops, gates that check fields no prompt asks for — the failure mode is: AI behaves roughly correctly; state never accumulates; user stays in Stage 4 forever; nobody knows why.

**Can a single prompt of this size keep all 8 stages in working memory?** Empirically yes, with Sonnet 4.6 (the model in `lib/journey/model.ts`). Claude routinely holds 200k-token contexts. The 3,200 tokens of static prompt + 500 tokens of state + 30 turns of history is comfortable. The issue isn't *holding* the stages in working memory; it's that the prompt currently doesn't *contain* enough of stages 3–8 to hold.

**Recommendation on the pivot:** keep it. Triple the depth of the per-stage move descriptions in `<method>`. Add an explicit `<stage_safeguards>` section that, for each stage, enumerates: (a) the named practices with required steps, (b) the prohibitions, (c) the closing rituals, (d) the exact readinessTouched tokens / state-report fields the AI must emit when each milestone fires. The master prompt grows to ~6,000 tokens — still well within budget, and the per-stage detail returns. The clinical canon is the human-reviewable parent; the master prompt becomes its dense LLM-target child.

The clinician-choosing-the-move framing is preserved. The safeguards are no longer optional.
