## Blockers cleared before this freeze

**PR #218** — `fix(commerce): stop Journey invoices touching MiniMind + block MiniMind double-buy`

- Journey installment invoices no longer zero MiniMind cycle counters or
  destroy top-up credits
- Journey installment payment failures no longer flip MiniMind
  subscribers to `free`
- MiniMind subscribers can't create a second subscription via direct POST
  (server 409 mirrors the /pricing UI hide)

**PR #219** — `feat(journey): enforce 1-year access + 5k anti-abuse cap; fix Purchase cascade` (schema change)

- `Purchase.user` now cascades on delete — Clerk `user.deleted` webhook
  and GDPR retention cron work for paying users
- 1-year Journey access enforced via new `Purchase.firstAccessedAt`
  column, stamped on the first accepted turn
- Anti-abuse ceiling of 5,000 messages per Purchase (approx. 12× a heavy
  engager's realistic use of the 8-stage arc)
- Single access helper at `lib/journey/access.ts`

**PR #220** — `fix(retention): delete Clerk user in nightly cron — close hard-delete resurrection loop`

- Retention cron calls `clerkClient.users.deleteUser` after DB delete —
  the deleted user's Clerk session is invalidated so the defensive
  upsert at `/home` can no longer resurrect them with fabricated consent

## Deferred to post-launch

Listed in the four audit reports; explicitly out of scope for this freeze:

- **E4** — Server-side consent capture via Clerk `unsafeMetadata`.
  Low actual risk while sign-up flows only through our checkbox-gated
  UI. Would need SignUp widget metadata + webhook read-back +
  grandfather logic for existing users.
- **Journey installment access-revocation on cancel.** Spec says
  "access ends at the last paid month"; code doesn't enforce. Bounded
  by the 1-year window landed in #219, so upper-bound loss is fixed.
- **Discharge / completion user-facing state.** Router sets
  `dischargedAt` but the client doesn't render a completion view.
- **Journey cost math nits.** `cancel_at` anchored to `Date.now()` not
  `subscription.start_date`; `stripe.subscriptions.update({cancel_at})`
  failure is silently caught.
- **Stripe Customer Portal not configured** — user could cancel the
  wrong subscription. Owner to configure portal defaults in Stripe
  dashboard.
- **Consent stamps on defensive upserts are `new Date()`, not real
  timestamps** (see E4 above).
- Assorted smaller nits — see the four audit reports in the session log.
